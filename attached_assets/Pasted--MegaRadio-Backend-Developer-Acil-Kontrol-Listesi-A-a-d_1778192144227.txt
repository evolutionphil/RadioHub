🔴 P0 — KRİTİK GÜVENLİK AÇIĞI
Sorun: Receipt validation çalışmıyor — fake receipt'ler premium yapıyor
Test: testuser2026@megaradio.test hesabı ile aşağıdaki request'i attım:
curl -X POST "https://api.themegaradio.com/api/user/subscription" \
  -H "Authorization: Bearer <valid_token>" \
  -H "X-API-Key: mr_VUzdIUHuXaagvWUC208Vzi_3lqEV1Vzw" \
  -H "Content-Type: application/json" \
  -d '{
    "platform":"ios",
    "productId":"megaradio_premium_yearly",
    "transactionId":"test_ios_tx_001",
    "originalTransactionId":"test_ios_tx_001",
    "receipt":"FAKE_BASE64_RECEIPT_FOR_TEST",
    "isTrial":false
  }'
Yanıt (HTTP 200):
{
  "success": true,
  "plan": "premium_yearly",
  "expiryDate": "2027-05-07T22:08:33.915Z",
  "isActive": true,
  "features": ["remove_ads","song_info","spotify_link","youtube_link","hd_stream","song_history","stream_record"]
}
⚠️ Backend, "FAKE_BASE64_RECEIPT_FOR_TEST" string'i ile gönderilen request'i kabul etti ve kullanıcıyı 1 yıllık premium yaptı. Bu, backend'in Apple'ın receipt validation servisine sorgu atmadığı, gönderilen receipt'i blindly güvendiği anlamına geliyor. Herkes kendi token'ını alıp curl ile sahte receipt göndererek ücretsiz premium olabilir.
Çözüm — Apple Receipt Validation (zorunlu)
iOS + Mac App Store için Apple'ın verifyReceipt endpoint'ine POST atılmalı:
Production: https://buy.itunes.apple.com/verifyReceipt
Sandbox (TestFlight): https://sandbox.itunes.apple.com/verifyReceipt
Best practice: Önce production'a POST at; eğer status == 21007 dönerse sandbox'a yönlendir.
Apple'a gönderilecek payload:
{
  "receipt-data": "<base64 receipt from client>",
  "password": "<App Store Connect Shared Secret>",
  "exclude-old-transactions": true
}
Shared Secret App Store Connect'ten alınır: App Store Connect → Apps → MegaRadio → App Information → App-Specific Shared Secret veya Master Shared Secret
Apple'ın yanıtında doğrulanacak alanlar:
status == 0 (geçerli receipt)
receipt.bundle_id == com.visiongo.megaradio
latest_receipt_info[].product_id == client'ın söylediği productId ile eşleşmeli
latest_receipt_info[].expires_date_ms > Date.now() (subscription için)
latest_receipt_info[].original_transaction_id ile DB'de duplicate kontrolü yap
Daha modern alternatif: StoreKit V2 (önerilen)
Apple Şubat 2024'ten beri StoreKit V2 + JWS imzalı transaction'ları öneriyor:
Endpoint: https://api.storekit.itunes.apple.com/inApps/v1/transactions/{transactionId}
Auth: JWT (ES256) — App Store Connect API Key (.p8) ile imzalı
Receipt yerine signedTransaction JWS string'i alır, kendi public key'iyle doğrular
Ne kullanılırsa kullanılsın, client'tan gelen veriyi mutlaka Apple'a doğrulattırmak şart.
🔴 P0 — Mac App Store platformu reddediliyor
Sorun
curl -X POST "https://api.themegaradio.com/api/user/subscription" \
  -H "Authorization: Bearer <token>" \
  -d '{"platform":"mac","productId":"megaradio_premium_yearly",...}'
Yanıt: HTTP 400 — {"error":"platform must be ios or android"}
Çözüm
Validation whitelist'e mac ekle:
const ALLOWED_PLATFORMS = ['ios', 'android', 'mac'];
if (!ALLOWED_PLATFORMS.includes(platform)) {
  return res.status(400).json({ error: "platform must be ios, android or mac" });
}
Mac receipt validation farkı
Mac App Store receipt'inin iOS receipt ile aynı format olduğunu unutma:
Aynı verifyReceipt endpoint'ine atılır
Aynı bundle_id (com.visiongo.megaradio) — Universal Purchase ayarlandığı için
Aynı product_id'ler kullanılır (megaradio_premium_yearly, _lifetime, _monthly1, megaradio_remove_ads_yearly1)
Receipt dosya yolu cihazda farklı (Contents/_MASReceipt/receipt) ama base64 encode edilince fark yok
Tek değişiklik: Validation kodunda platform === 'mac' dalı ios ile aynı işlemi yapmalı

🟡 P1 — Apple Server-to-Server Notifications (App Store Server Notifications V2)
Auto-renewable subscription'lar için Apple, kullanıcı yenilediğinde / iptal ettiğinde / refund aldığında backend'e webhook gönderir. Bu kurulmazsa:
Kullanıcı App Store'da iptal ederse → backend bunu öğrenmez → uygulama kullanıcıya hâlâ premium gösterir.
Apple refund verirse → kullanıcı parayı geri alır ama premium hâlâ aktif kalır.
Subscription expire olur → backend manuel kontrol etmediği için kullanıcı sonsuza kadar premium görünür.
Kurulum
App Store Connect → Apps → MegaRadio → App Information → App Store Server Notifications V2 URL
URL: https://api.themegaradio.com/api/iap/apple-webhook
Production + Sandbox için ayrı URL'ler verilebilir (önerilir).
Webhook payload (özet)
Apple, JWT (ES256) imzalı bir notification gönderir. İçinde notificationType alanı var:
notificationType	Yapılacak işlem
SUBSCRIBED	Yeni premium kaydı
DID_RENEW	expiryDate'i güncelle
DID_CHANGE_RENEWAL_STATUS	Auto-renew on/off bilgisini sakla
EXPIRED	isActive = false
REFUND	Premium'u iptal et + kullanıcıya bildirim gönder
REVOKE	Family sharing iptali — premium'u kaldır
JWS imzasını mutlaka doğrula (Apple'ın public key'leri ile).
🟢 P2 — Google Play Developer API entegrasyonu (Android için)
iOS/Mac doğrulanırken Android purchaseToken'ı da Google Play Developer API'ye sorulmalı:
GET https://androidpublisher.googleapis.com/androidpublisher/v3/applications/{packageName}/purchases/subscriptions/{subscriptionId}/tokens/{token}
Authorization: Bearer <google_oauth_token>
Aynı şekilde sahte purchaseToken'larla premium alınmasını engellemek için bu validation şart.
